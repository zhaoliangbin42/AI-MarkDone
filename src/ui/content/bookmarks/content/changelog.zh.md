## 原理解析

又到了我最喜欢的原理环节。

### 为什么有些划词翻译软件会失效？

之前有朋友反馈，开启 AI-MarkDone 后，网页上的划词翻译软件可能无法正常弹出。

这个问题比较特殊，因为不同划词翻译插件的实现方式并不完全一样。我们的推测是，很多划词翻译工具都会监听网页中的文字选择、鼠标事件和焦点变化，然后在选区附近插入自己的浮层。而之前 AI-MarkDone 会在 ChatGPT 官方顶部区域中插入一个书签入口，这意味着多个插件可能同时修改或监听同一片比较敏感的页面区域。

这种情况下，即使双方都没有主动阻止对方，也可能因为页面结构、事件顺序、焦点或者浮层判断发生变化，最终导致划词翻译弹窗无法正常出现。

这次我们换了一个更克制的做法：不再向 ChatGPT 官方顶部区域插入入口，而是把书签面板入口移动到右下角由 AI-MarkDone 自己管理的操作区。这样可以尽量减少对 ChatGPT 官方页面结构的修改，也降低与划词翻译、文本选择等第三方插件发生冲突的可能。

因为划词翻译插件很多，我们无法确认所有插件失效的内部原因都完全相同，所以这里仍然只是基于现象和实现方式作出的推测。不过从插件边界上看，减少对官方页面敏感区域的侵入，是目前更稳妥、更通用的解决方式。

### 阅读器注释为什么需要重新整理？

这一版还要特别感谢 GitHub 用户 @rqhu1995 提交代码，改进了阅读器的注释管理。

过去的注释功能更偏向“边读边标”：选中一段内容、添加注释，然后通过正文高亮和右侧锚点重新找到它。这个方式适合少量注释，但当一条长消息里积累了很多注释之后，只靠正文中的标记就不太方便了。

这次改进增加了统一的注释列表，可以按照创建时间或者正文位置排序，也可以从列表中定位、编辑和删除注释。注释在复制、导出和重新插入发送框时，也会沿用同一套排序规则。

这里面比较值得讲的一点，是“正文位置”并不等于屏幕上的像素位置。页面会因为字体、窗口宽度和渲染状态发生变化，所以不能简单记住某个坐标。更可靠的方式是记录注释对应的文本区间和上下文，再根据当前正文重新定位。这样即使阅读器重新打开、宽度发生变化，注释仍然有机会回到正确的内容附近。

另外，LaTeX 和 TeX 代码块现在也会默认使用更适合阅读的换行方式，并且可以手动切换。对于公式源码比较长的内容，不再需要一直横向滚动。

再次感谢 @rqhu1995 的贡献。也欢迎更多朋友通过 GitHub 提交 Issue 或 PR。

## Prompts 面板

回到 AI-MarkDone。这次我最想拿出来聊的，是新的 Prompts 面板。

其实很多朋友应该已经装过 Prompt 管理类插件。一开始我也觉得，AI-MarkDone 没必要再做一个同类工具。但后来想想，如果为了 A 功能装一个插件，为了 B 功能又装一个插件，最后浏览器里会越来越乱。既然 AI-MarkDone 已经在做长对话阅读、复制、整理和发送相关的工作，那一些高频 Prompt 能力放进来，也说得通。

所以这一版增加了一个本地 Prompt 管理面板。它不是云端服务，也不是 Prompt 市场，只是一个很轻量的本地工具。你可以维护自己的常用 Prompt，也可以在 ChatGPT 官方输入框和阅读器发送面板里输入 `\` 快速唤起候选。

我还内置了几个默认 Prompt，主要是想给新手朋友一点思路。老手可以直接跳过或删掉。

其中我最想聊的是两个：

1. 网页端 ChatGPT 也可以“借用”传说中的 Skills。比如 `Humanize Text With a Skill` 这个 Prompt，示例思路就是把 GitHub 上的 Skill 链接直接交给 ChatGPT，让它先参考那份 Skill 说明，再按里面的方法完成文本润色。
2. 既然网页端调用 Skills 不够方便，那为什么不直接把 Skills 封装成 Prompt 呢？这是我自己思考 Skills 本质时冒出来的一个想法。`Create a Reusable Skill` 就是基于这个想法做的小实验：你告诉 ChatGPT 想做一个什么样的 Skill，再补充一些流程、步骤和约束，它会尝试深入理解这些流程，然后把整个 Skill 拉平成一个完整 Prompt 输出给你。这里面可能包括对应的指令，也可能包括可以执行的脚本，本质上就是一个 Skill 的扁平化形式。基本用法是：先用这个 Prompt 生成你想要的 Skill Prompt，再把生成出来的 Prompt 复制到一个新对话里执行。理论上，它就会按这个 Skill 的方式继续工作。

当然，这个想法还不算成熟，我自己也没有做特别充分的测试。大家可以把它当成一个小实验玩一玩。它不一定每次都好用，但我希望它至少能带来一点思考：Prompt 不只是几句话，它也可以是一套流程、一组约束，甚至是一种小型工作方法。

当然，Prompts 模块仍然保持 AI-MarkDone 一贯的原则：尽量轻，不打扰。你可以通过右下角入口打开，也可以用 `\` 快速插入；如果不需要，也可以在设置里关闭右下角入口，或者单独关闭输入 `\` 时自动出现的 Prompt 联想。

另外，我也会在小红书上继续分享一些 ChatGPT 使用经验，以及我对 AI 使用方式的思考。也欢迎大家多点一下插件图标，每次更新都会有更新日志，可以及时看到新功能和一些背后的想法。

# 4.6.0
2026-06-18

大家好啊，这次更新酝酿了很久，也做了一次比较大幅度的升级。

这段时间我一直想解决一个很具体、也很烦人的问题：ChatGPT 长对话真的太容易卡了。我自己是 Pro 用户，用 Pro 模型时，每一条消息都很长。实际用下来，超过三条长消息之后，网页就已经明显卡得动不了了；哪怕打开阅读器，也只能缓解，不能从根本上解决。

从原理上讲，问题并不神秘。ChatGPT 页面本身要渲染非常重的 Markdown，尤其是公式很多的时候，渲染压力会非常大。只要你还在 ChatGPT 页面里操作，不管是滚动、点击、复制，还是打开插件界面，都多少会被这个页面的渲染压力拖住。ChatGPT 官方其实也做过优化。大概从 4 月底开始，网页逐渐变成增量加载模式，旧消息会被适当卸载，以降低整体渲染压力。但即便如此，长消息本身带来的渲染压力还是没有办法彻底消失。

所以这次我换了一个思路：既然 ChatGPT 页面本身卡，那能不能跳出这个页面？

## 原理解析

又到了我最喜欢的原理环节。

AI-MarkDone 原本就有阅读器能力：在 ChatGPT 页面中发现消息、提取内容、整理结构，然后在阅读器里重新渲染。既然已经能提取消息，那么我就在想，能不能把这份阅读器内容放到一个独立的扩展页里打开？

查了一圈浏览器扩展文档之后，我发现这是可行的。扩展页可以通过浏览器 runtime 和原始 ChatGPT 页面建立桥接。也就是说，独立页负责渲染一个完整的阅读器，原 ChatGPT 页面负责提供内容、发送、刷新、定位等能力；两边通过浏览器内部通信保持连接。

这样一来，阅读本身就不再受 ChatGPT 官方网页的 DOM 和 Markdown 渲染压力影响。独立阅读器跑在扩展自己的页面里，页面更干净，渲染压力也更可控。你仍然可以刷新、发送、定位原文，但主要阅读体验已经从卡顿的 ChatGPT 页面里抽离出来了。

当然，真正做起来并不是把阅读器搬过去这么简单。因为涉及原页面和独立页之间的桥接、会话、刷新、发送、定位、书签、主题、公式样式等一整套对齐，所以这次重写和补齐了不少底层链路。既然都叫阅读器了，我也顺手把一些原本藏在设置里的阅读功能搬回阅读器内部：现在可以直接调节字号、宽度和显示方式。

说实话，我现在自己更喜欢在阅读器里继续对话。它既能避开长网页卡顿，又有注释、Sticky、复制、书签等一堆顺手的小功能。还没试过阅读器的朋友，尤其是经常读长对话的朋友，真的建议试试。

## 隐私说明

这里也必须认真说明一下隐私问题。

AI-MarkDone 当前没有任何联网功能，代码也全部公开在 GitHub 上。如果你不放心，可以把 GitHub 链接发给 ChatGPT，让它帮你一起看代码。

插件里的内容发现链路，本质上是为了复制 Markdown、导航栏、阅读器、书签和导出这些功能服务的。只有能读取当前页面里的内容，才可能支撑这些能力。新的独立阅读器也是一样：它是在浏览器管理的扩展页里运行，通过浏览器内部通道和原 ChatGPT 标签页通信，不会把内容传到任何外部服务器。关闭独立页后，这次打开的会话也就结束了，不会额外长期保存。

## 原理解析

又到了我最喜欢的原理解析环节。我很喜欢把开发过程中发现的问题、学到的知识和总结出来的经验分享给大家，所以这个栏目后续会尽量稳定保留下来。

这次想聊的是一个很烦人的问题：ChatGPT 网页开太久之后，插件有时候会像“失联”了一样，点按钮没反应，或者后台出现找不到 tab 的报错。

这个问题和 Chrome 扩展的运行机制有关。现在 Chrome 扩展大多运行在 MV3 机制下，后台不是一个一直常驻的页面，而是一个事件驱动的 service worker。也就是说，浏览器觉得它暂时没事做的时候，就可能把它停掉；等下一次有事件发生，再把它唤醒。

同时，如果一个 ChatGPT 标签页长时间放在后台，浏览器也可能对它进行冻结、丢弃或延迟恢复。这个时候，后台想给某个 tab 发消息，可能会发现 tab 已经关了、id 过期了，或者页面里的 content script 暂时还没恢复，于是就会出现类似“找不到这个 tab”或者“接收端不存在”的错误。

这次的思路不是强行让后台一直活着。那样虽然看起来简单，但会更耗资源，也不符合浏览器现在的设计方向。我们做的是更轻量的恢复链路：插件在和页面通信前，会先用 ping 确认当前页面是否还连得上；如果连不上，就不硬发命令，也不把这种生命周期里的正常失败当成致命错误。等 ChatGPT 页面恢复之后，content script 会主动发一个 ready 信号给后台，告诉后台“我回来了”。后台再根据这个 tab 的真实 id 更新状态。

简单说，就是不常驻、不轮询、不强行保活，而是在浏览器把页面和后台唤醒之后，让它们重新握手。这个思路我还会继续观察和验证，但我觉得方向是对的：既要恢复得回来，也不要为了恢复而偷偷消耗系统资源。

## 原理解析

这次性能优化的重点不是把页面“刷得更快”，而是尽量减少不必要的刷新。ChatGPT 页面本身会不断增量加载、替换节点、补齐工具栏和调整布局，如果插件把每一次 DOM 变化都当成“整页重新分析”的信号，就很容易在长对话里产生额外卡顿。

这一版中，目录条的刷新逻辑会更谨慎地判断变化是否真的影响对话结构；如果只是局部按钮、工具栏或无关节点发生变化，就尽量不触发完整重建。这样在打开长对话、等待消息继续生成，或者 ChatGPT 自己补齐页面元素的时候，目录条不需要反复做同一件事。

工具栏和页面宽度变化也做了类似处理。拖动窗口宽度时，页面会连续触发很多布局变化；以前这些变化可能会和工具栏、目录条的渲染互相干扰。现在会先让页面宽度变化稳定下来，再恢复相关 UI 的刷新，减少拖动窗口时的抖动和重复计算。

阅读器的新页面刷新逻辑也一起升级了。现在在阅读器中继续发送消息后，新增内容会更稳定地进入阅读器页面，不需要反复关闭再打开。整体目标是让插件在 ChatGPT 这种动态页面里少打扰、多等待、只在真正需要的时候更新。

# 4.4.0
2026-05-12

感谢 GitHub 用户 @LTong-g 贡献代码，这一版本针对“将消息导出为 PNG”功能做了大幅度优化。

## 原理分享

之前导出为 PNG 的基本思路，是将内容渲染之后按照高度分块。但是实际上，对于包含大量公式、代码块、表格或图片的内容块来说，即使它的视觉高度不一定特别夸张，背后的 DOM 结构也可能非常复杂，单块渲染压力依旧巨大。@LTong-g 的做法是：除了按高度分块外，也会按 DOM 复杂度分块，从而减少复杂消息导致的长时间浏览器卡顿。

这一版也给 PNG 导出补上了进度反馈。复制为 PNG 时会显示可取消的进度面板；批量导出 PNG 时会同时显示当前消息进度和总进度。导出需要等待时，界面应该告诉你它正在做什么，而不是让按钮看起来像卡住了。

欢迎大家反馈或者贡献代码。我希望每次更新都能将原理描述一下，满足大家对原理的好奇心，也激发大家参与进来的热情。

## 变更
- 提升导出 PNG 的性能，并添加进度条，缓解等待焦虑（感谢 GitHub 用户 @LTong-g 贡献代码）。
- 更新内容发现链路。ChatGPT 最近网页元素有所变动，旧的内容发现链路只能探索到最近几条消息，导致阅读器和导出功能都受限，现在问题已修复（感谢小红书用户 @Jim 和 @全是恶意）。

# 4.2.0

当然，这几个平台也有替代插件，欢迎大家下载使用。

对于 Gemini，目前做得最好的插件应该是 [Gemini Voyager](https://github.com/Nagi-ovo/gemini-voyager)。

对于全平台书签功能，我之前用过一个非常喜欢的插件是 [Timeline](https://github.com/houyanchao/Timeline)。

这次更新还增加了两个非常实用的功能，分别是导出消息为图片，以及导航栏的展开模式。其中，导出消息为图片是我个人比较喜欢的功能。平时，ChatGPT 消息分享主要还是以链接分享或者截图分享为主，但是对于很多内容比较长的对话，就需要用到长截图，上手难度会大幅增加。因此，在充分考虑用户分享场景后，我给插件添加了一键复制消息为图片以及批量导出消息为图片这两个功能。

另外，之前导航栏调的手风琴效果，好看是好看，但实用性欠佳。因为如果窗口中有几十条消息，我们无法一眼定位到想要的那条消息，而是要来回滑动去寻找。因此这个版本中引入了导航栏的全展开模式，方便用户一眼定位消息。

## 新增
- 新增对 Safari 浏览器和 Firefox 浏览器的支持。

- 修复导航栏点击一次无法正确定位到目标消息的问题（感谢小红书用户 @谦卑于世）。

## 原理解析
- 之前导航栏点击一次之后，会跳转到对应消息所在的位置。但是这个时候会触发 ChatGPT 官方的增量加载，而加载出来的内容会导致布局有所偏移，进而导致定位也发生偏差。所以之前导航栏需要点击第一次并跳转到对应位置之后，等网页稳定下来，再点击一次才能到达目标位置。现在的方案是点击一次之后，会先尝试跳转到对应的位置，然后监测增量加载之后的坐标变化，再进行一次跳转。第二次跳转对用户来说几乎无感。

# 4.1.2
2026-04-22

2026年4月22日凌晨，ChatGPT进行了一次较大的版本更新，导致页面内信息的全量加载变成了增量加载，原有的信息识别算法全部失效，影响范围包括但不限于：折叠条功能直接失效、阅读器只能识别到最终渲染的4~5条消息等。

鉴于最初开发折叠条的目的是为了提高页面的加载速度，降低渲染压力，如今 ChatGPT 已经官方支持增量加载，会自动卸载非活跃节点，性能得到大幅提升。另外也由于 ChatGPT 的动态加载方式，导致折叠条的开发复杂度大幅提高，稳定性大幅降低。经过综合评估，决定将折叠条功能删除。

关于阅读器，由于当前网页默认只加载视窗内的4~5条消息，以往获取页面内稳定节点的方法已经失效，阅读器无法一次性将所有内容全部展示。为了解决这个问题，针对 ChatGPT 重新设计了内容发现引擎，现在已经具备进入页面之后，阅读器能识别所有消息的能力。

对于官方的浏览页面，经过我的实际使用，我感觉这次更新后的动态加载也有一些不太顺手的地方。比如滚动条快速拉动时，内容显示会有延迟，导致旧消息翻看难度提升。因此，为了照顾喜欢使用原生页面的用户，我顺带将当前的底层能力提取出来做成了目录。进入对话页后，目录将会在右侧展示，支持实时消息预览，并且点击之后能够直接跳转到想要的位置。

# 4.1.0
2026-04-19

这一版的重点有三件事。第一，独立的源码复制面板被删掉了，源码复制能力直接并进了阅读器。第二，新增了“灵动注释”，可以边读边标、边写修改意见，再一次整理成适合发给大模型的输入。第三，书签面板补上了 Changelog、About 和 FAQ 三个信息页，方便查看版本变化、项目背景和常见问题。

其中，把源码复制直接搬到阅读器里，是这次最难做的一部分。公式、代码块、表格、图片这类内容，并不是普通文本，它们更像一个个需要整体处理的闭合单元。如果直接沿用浏览器默认复制，最后拿到的往往只是渲染后的可见文本，原来的 Markdown 结构很容易丢掉。现在的做法，是在阅读器里先识别这些闭合单元对应的源码边界；当你选中它们时，复制逻辑会优先回到原始 Markdown 片段，再按选区顺序重新拼起来，所以复制结果会尽量保持标准 Markdown，而不只是页面上看见的那层文本。

## 新增
- 阅读器里的源码复制现在直接可用，不再需要单独打开源码面板。
- 阅读器支持对代码、公式、表格、图片等闭合内容按源码进行选择和复制，复制结果会尽量保持标准 Markdown 形式。
- 阅读器现在支持对选中内容直接添加灵动注释。选中正文后，会在附近出现悬浮的复制和注释按钮。
- 注释会和对应选区建立关联，并在阅读器中以高亮方式持续标记出来；已有注释也支持再次打开查看和编辑。
- 新增注释导出模板，可在“设置 > 阅读器 > 注释复制模板”中自定义每条注释的导出格式。
- 新增 Prompt Library，可在“设置 > 阅读器 > Prompts”中维护多条可复用提示词。
- 注释结果支持一键复制，也可以直接插入阅读器左下角的发送消息框上方，减少手动复制粘贴。
- 书签面板新增 Changelog、About 和 FAQ 信息页，帮助你更快了解版本变化和项目背景。
